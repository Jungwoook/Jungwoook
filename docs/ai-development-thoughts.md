# AI를 활용하는 개발에 대한 생각

> AI는 단순 코드 생성 도구가 아니라,
>
> 개발자의 설계와 검증 과정을 보조하는 협업 도구라고 생각합니다.

## 1. AI를 개발에 활용하게 된 배경

이미 개발 영역에서 AI는 더 이상 빼놓을 수 없는 존재가 되었습니다.

AI는 이제 단순 코드 생성 도구를 넘어,
설계, 문서화, 테스트, 리뷰 등 개발 전반을 보조할 수 있는 도구가 되었다고 생각합니다.

하지만 실제로 활용해보면서,
단순 요청만으로는 프로젝트 방향성과 일관성을 유지하기 어렵다는 점을 경험했습니다.

특히 프로젝트 규모가 커질수록, 기존 구조와 맞지 않는 구현이 추가되거나,
불필요하게 복잡한 코드가 반복되는 경우가 많았습니다.

결과적으로 코드는 동작하더라도,
시간이 지날수록 프로젝트를 파악하기 힘들고, 유지보수가 어려워지는 상황을 자주 경험했습니다.

## 2. 문서 기반 개발을 중요하게 생각하는 이유

AI는 빠르게 결과물을 생성하지만, 명확한 기준 없이 프로젝트를 진행한다면 프로젝트의 전체 구조나 이전 작업의 맥락까지 일관되게 유지하지는 못하는 경향이 있습니다.

특히 프로젝트 규모가 커질수록, 이전 설계 의도와 다른 방향으로 구현되거나, 이미 구현된 구조가 반복해서 변경되는 경우가 많았습니다.

처음에는 요구사항을 기반으로 단순히 채팅으로 작업을 진행했지만,
작업 내용이 누적될수록 프로젝트 방향성과 구조를 지속적으로 설명해야 하는 상황이 반복되었습니다.

이러한 문제를 줄이기 위해,
프로젝트의 구조와 작업 흐름을 문서 형태로 정리하기 시작했습니다.

문서는 단순 기록 목적이 아니라,

프로젝트의 방향성과 설계 의도를 유지하고,
구현 과정에서 발생한 변경 사항과 작업 이유를 추적하며,
AI가 프로젝트의 작업 맥락을 이해할 수 있도록 만드는 역할을 한다고 생각합니다.

특히 프로젝트 규모가 커질수록,

- 왜 해당 구조를 선택했는지
- 어떤 이유로 구현 방식이 변경되었는지
- 어떤 리뷰와 검토 과정이 반영되었는지

를 명확하게 남기는 것이 중요하다고 생각하게 되었습니다.

## 3. AI 기반 개발에서 문서가 프로젝트에서 수행하는 역할

프로젝트를 진행하면서 문서는 단순 설계 기록 이상의 역할을 하게 되었습니다.

### 요구사항 문서

요구사항 문서는 프로젝트의 방향성을 정리하는 기준이 되었습니다.

초기에는 구현 중심으로 빠르게 작업을 진행했지만,
요구사항이 충분히 정리되지 않은 상태에서는
AI가 기능을 구현할 때마다 의도와 다른 결과가 반복해서 발생했습니다.

특히 같은 기능이라도:

- 어떤 조건에서 동작해야 하는지
- 어떤 예외 상황을 고려해야 하는지
- 어떤 방향으로 구현되어야 하는지

가 명확하지 않으면, 구현 결과가 요청마다 달라지는 경우가 많았습니다.

결과적으로 기능은 동작하더라도,
프로젝트 요구사항과 어긋나는 부분을 직접 다시 확인하고 수정해야 하는 작업이 반복되었습니다.

이러한 경험 이후에는
구현 이전에 요구사항과 동작 기준을 먼저 정리하는 과정을 중요하게 생각하게 되었습니다.

---

### 아키텍처 문서

아키텍처 문서는 구조 변경 과정에서 프로젝트의 방향성을 유지하는 기준이 되었습니다.

특히 초기에는 아키텍처가 충분히 정리되지 않은 상태에서 구현을 진행하면서,
기능을 추가할 때마다 구조가 반복해서 변경되는 경우가 많았습니다.

AI는 요청 단위로 구현을 진행하기 때문에,
프로젝트 전체 구조가 명확하게 정의되어 있지 않으면
이전 구현 방식과 다른 구조를 새롭게 제안하거나,
이미 존재하는 역할과 중복되는 코드가 추가되는 경우도 있었습니다.

결과적으로 기능은 동작하더라도,
구현이 반복될수록 구조 일관성이 점점 무너지는 상황을 경험하게 되었습니다.

이러한 경험 이후에는
구현 이전에 프로젝트 구조와 역할을 먼저 문서로 정리하고,
아키텍처 문서를 기준으로 구조 변경 여부를 검토하는 방식을 중요하게 생각하게 되었습니다.

---

### 작업 기록 문서

작업 기록 문서는 구현 과정에서 발생한 변경 이유와 작업 흐름을 추적하는 역할을 하게 되었습니다.

AI를 활용한 개발은 구현 속도가 빠르기 때문에,
짧은 시간 안에도 구조 변경과 기능 수정이 반복해서 발생하는 경우가 많았습니다.

하지만 시간이 지난 뒤 다시 코드를 확인했을 때는,

- 왜 해당 구조를 선택했는지
- 어떤 문제를 해결하기 위한 수정이었는지
- 어떤 리뷰 결과가 반영되었는지

를 코드만으로는 파악하기 어려운 경우가 많았습니다.

특히 구조 변경이 반복될수록,
이전 구현 의도와 수정 이유가 빠르게 누락되는 문제를 자주 경험했습니다.

이러한 이유로 작업 기록 문서는 단순 TODO 목록이 아니라,

- 어떤 문제가 있었는지
- 어떤 방향으로 구조를 변경했는지
- 어떤 의도를 기준으로 수정했는지
- 변경 이후 어떤 결과가 발생했는지

를 기록하는 문서로 활용하고 있습니다.

실제 작업 기록 문서에서는
문제 상황, 수정 방향, 구조 변경 내용, 결과 등을 함께 정리하며,
프로젝트의 변경 흐름과 의사결정 과정을 추적할 수 있도록 관리하고 있습니다.

## 4. 실제 프로젝트 활용

### GitHub 이슈관리 프로젝트

> GitHub/GitLab 기반 멀티 플랫폼 구조 설계를 진행하며,
>
> AI를 활용한 문서 기반 개발 프로세스를 적용하였습니다.

실제 프로젝트에서는 요구사항, 아키텍처, 작업 기록을 중심으로 문서를 분리하여 관리하고 있습니다.

이는 단순 문서 정리를 위한 목적이 아니라,

- 프로젝트 방향성을 유지하고
- 구조 변경 과정을 추적하며
- 구현 흐름과 리뷰 결과를 관리하기 위한 목적입니다.

특히 AI를 활용한 개발에서는 구현 속도가 빠른 만큼
구조 변경과 작업 맥락이 빠르게 누락되는 경우가 많았기 때문에,

프로젝트의 요구사항, 구조, 변경 이유를 문서 형태로 지속적으로 관리하는 방식을 중요하게 생각하게 되었습니다.

```bash
github-issue-manager/
├── AGENTS.md                          # AI 작업 규칙 및 개발 프로세스 정의
├── CLAUDE.md                          # Claude 코드 리뷰 기준 및 출력 규칙 정의
├── docs/
│   ├── 01-prd.md                          # 요구사항 정의 및 프로젝트 방향성 관리
│   ├── 03-architecture.md                 # 전체 아키텍처 구조 및 역할 정의
│   ├── 04-architecture-transition-*.md    # 구조 변경 과정 및 아키텍처 개선 이력
│   ├── tasks/                             # 작업 단위 문서 및 구현 과정 기록
│   ├── reviews/                           # AI 및 코드 리뷰 문서
│   ├── review-results/                    # 리뷰 결과 및 반영 내역 관리
│   ├── openapi/                           # OpenAPI 및 API 문서
│   ├── guides/                            # 개발 및 작업 가이드 문서
│   └── archive/                           # 이전 설계 및 작업 이력 보관
│
├── backend/                           # Spring Boot 백엔드 애플리케이션
├── frontend/                          # React 프론트엔드 애플리케이션
└── ...
```

## 5. AI개발에서 개발자의 역할

AI 개발 시대에서 가장 중요한 역량은,
AI가 생성한 결과물을 이해하고 검증할 수 있는 개발 역량이라고 생각합니다.

AI는 빠르게 결과물을 생성할 수 있지만,
프로젝트의 방향성과 유지보수 관점까지 항상 고려하는 것은 아니라고 느꼈습니다.

실제로 개발을 진행하면서도,
기능은 동작하더라도 프로젝트 구조와 맞지 않거나,
장기적으로 유지보수가 어려운 방식으로 구현되는 경우를 자주 경험했습니다.

특히 새로운 기술이나 구조를 제안받았을 때,
단순히 적용하는 것이 아니라,

- 왜 해당 기술이 필요한지
- 현재 프로젝트 요구사항에 적합한 구조인지
- 다른 방식과 비교했을 때 어떤 장단점이 있는지

를 직접 이해하고 판단하는 과정이 중요하다고 생각하게 되었습니다.

결국 AI를 활용하더라도,
프로젝트의 방향성과 구조를 유지하는 역할은 여전히 개발자에게 중요하다고 생각합니다.

이를 위해서는 단순히 AI 도구를 사용하는 것보다,

- CS 지식
- 설계 이해
- 유지보수 관점
- 실제 구현 경험

을 기반으로 결과물을 검토하고 판단할 수 있는 역량이 필요하다고 생각합니다.

또한 AI를 활용한 개발 환경 역시 계속해서 변화하고 있습니다.

저 역시 채팅형 AI뿐만 아니라,
문서 기반 개발, 구현과 리뷰 분리, GUI/CLI 기반 도구 활용 등
다양한 방식으로 개발 흐름을 계속 개선해오고 있습니다.

앞으로도 단순히 AI를 사용하는 것에 그치지 않고,
프로젝트의 구조와 변경 흐름을 직접 이해하고 관리할 수 있는 개발 방식을 계속 고민해나가고자 합니다.
